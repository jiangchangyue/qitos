---
title: "为什么 QitOS 坚持单一内核"
description: "为什么 QitOS 的示例、基准测试和生产型智能体都运行在同一个 AgentModule + Engine 内核上。"
---

QitOS 最强的一条设计主张，也许同时也是最简单的一条：

**一次运行应该只有一个执行内核。**

在 QitOS 里，具体来说：

- `AgentModule` 定义策略
- `Engine` 负责执行
- 解析器、工具集、评估器、记忆与追踪都附着在这条主链路上

我们会避免为了规划、基准测试或领域工作流再额外引入一层隐藏运行时。

## 为什么这很重要

一旦框架里出现多个隐式运行时，通常会有三个后果：

1. 示例和生产型智能体变得不可比
2. 基准测试逻辑和日常智能体逻辑逐渐漂移
3. 追踪记录不再能讲清楚完整的执行故事

QitOS 就是为了避免这三件事。

如果基准测试运行器、Claude Code 风格智能体和安全审计智能体全都共享同一个执行内核，那么：

- 追踪记录可以直接横向比较
- 调试方法可以复用
- 提示词与解析器实验更容易解释
- 长时运行行为会保持可检查，而不会退化成框架胶水

## 专门化应该放在哪里

QitOS 并不反对专门化，但它希望专门化落在正确的位置：

- 状态设计
- 提示词策略
- 解析器与协议选择
- 工具组合
- `reduce()` 语义

这也是为什么领域化安全审计智能体仍然可以是一个普通的 QitOS 智能体。

## 对用户来说

如果你正在基于 QitOS 设计智能体，默认问题通常不该是：

> 我是不是要额外做一个编排器？

而应该更常问：

> 我能不能在同一个内核之上，通过状态、提示词、解析器、工具、历史或记忆表达这件事？

这种约束会让系统更易学、更易测，也更易复现。
