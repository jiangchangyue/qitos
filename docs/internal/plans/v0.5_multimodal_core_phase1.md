# v0.5 Multimodal Core Phase 1

Status: implemented on top of the v0.3 runtime kernel.

## Goal

Ship the first clean multimodal core for QitOS without breaking the existing text-first agent path.

This phase intentionally focuses on:

- OpenAI-compatible `chat.completions` multimodal input
- screenshot-first observation handling
- generic GUI / multimodal SDK primitives
- trace/qita visibility for visual assets

This phase intentionally does **not** yet promise:

- a full OSWorld runner
- complete GUI action execution
- screenshot timeline replay overlays
- provider-wide multimodal parity

## Delivered

### 1. Multimodal core types

Added lightweight core types and helpers in `qitos.core.multimodal`:

- `ContentBlock`
- `MessageEnvelope`
- `ObservationPack`
- `GroundingMetadata`
- `VisualTraceAsset`

These provide one normalized vocabulary for image input and visual observations.

### 2. Model input upgrade

Upgraded the base model message contract from text-only assumptions toward mixed-content messages:

- legacy text messages still work unchanged
- multimodal content arrays now normalize through one shared helper path
- OpenAI-compatible adapters now map image blocks onto chat-completions `content` arrays

### 3. OpenAI-compatible vision path

Implemented first-phase support for:

- image URL blocks
- image base64 / data URL blocks
- local image file blocks

Pure-text OpenAI-compatible requests still keep their previous wire shape.

### 4. Engine multimodal wiring

The Engine now knows how to build a current-step multimodal user turn from:

- prepared task text
- prompt-side user content blocks
- image task resources
- multimodal environment observations

Current policy is intentionally conservative:

- only the current step's visual input is injected
- history remains text-first for now

### 5. Environment SDK foundation

Added multimodal / GUI-oriented contracts:

- `GUIObserverCapability`
- `GUIControllerCapability`
- `OCRCapability`
- `GroundingCapability`

Added a minimal `ScreenshotEnv` so we can validate the multimodal path end-to-end without binding the abstraction to one benchmark.

### 6. Trace and qita visibility

Trace steps now record:

- `visual_assets`
- `observation_modalities`
- `visual_asset_count`
- `has_screenshot`
- `has_dom`
- `has_accessibility_tree`
- `model_input_modalities`
- `model_input_visual_count`

`qita` now exposes a first-pass visual-assets section and can serve local screenshot assets for run inspection.

### 7. Example baseline

Added `examples/real/visual_inspect_agent.py` as the first minimal visual baseline.

The example demonstrates:

- screenshot-first multimodal input
- OpenAI-compatible model usage
- existing parser/protocol compatibility
- deterministic smoke execution

## Follow-up for Phase 2

The next logical steps are:

1. add richer visual-web examples combining screenshot + DOM + OCR
2. add a dedicated OSWorld-style GUI adapter package on top of the new env contracts
3. add capability-aware multimodal fallbacks for more model families
4. upgrade qita from asset display to true screenshot timeline / visual replay
5. introduce explicit GUI action schemas and controller adapters beyond no-op/mock control
