# v0.7 Handoff Scope

## What is in v0.6 scope (DelegateTool only)

- `DelegateTool` with nested `Engine.run()` — synchronous delegation
- `AgentSpec` with `model_override`, `tools_override` fields
- `AgentRegistry.get_delegate_tools()` and `get_handoff_tools()`
- `DelegateEventInterceptor` for first-class DELEGATE_START/END events
- Sub-trace writer with depth-aware run_id
- Depth guard: `MAX_DELEGATE_DEPTH=3`

## What is deferred to v0.7

1. **Decision.handoff() Engine loop modification** — The handoff Decision-mode already exists but the Engine loop handling is basic. v0.7 needs:
   - Full context strategy implementation (FULL/SUMMARY/ISOLATED) for handoff
   - Shared memory injection on handoff
   - State adapter invocation during agent switching

2. **HandoffTool as first-class tool** — Currently HandoffTool is a signal-only tool. v0.7 should wire it into the Engine's action execution pipeline so that calling `transfer_to_{agent}` triggers the same handoff flow as `Decision.handoff()`.

3. **Canonical multi-agent templates** — Manager-Worker, Planner-Executor, Proposer-Verifier patterns using the full handoff API.

4. **qita handoff timeline** — Visual timeline showing handoff flow between agents in the run page.

5. **Async delegate** — Support for asynchronous delegation (requires event loop integration).

## Key design decisions

- DelegateTool creates a nested `Engine.run()` which is synchronous and blocking. The parent agent waits for the sub-agent to complete.
- Handoff (Decision mode) swaps the agent in the same Engine loop. This requires Engine loop changes and is higher risk.
- The Engine's handoff handling at line ~1009 is stable for v0.6 and should not be modified.
